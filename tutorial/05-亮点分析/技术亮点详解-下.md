## 亮点六：领券并发优化 — LUA脚本原子校验 + MQ异步写库

### 6.1 设计背景/问题

优惠券领取是典型的高并发场景（类似秒杀），核心挑战：
- **超发问题**：库存扣减必须原子性，否则多线程并发导致超发
- **限领问题**：每人限领N张，校验+扣减不是原子操作会导致超领
- **性能问题**：如果每次领券都访问数据库，数据库扛不住高并发

### 6.2 解决方案

采用 **Redis LUA脚本** 实现原子性校验（库存+限领+时间），校验通过后通过 **MQ异步** 写入数据库，实现读写分离。

```
前端请求 → LUA脚本原子校验(Redis) → 发送MQ消息 → 消费者异步写MySQL
```

### 6.3 核心代码分析

#### 6.3.1 领券LUA脚本

> 源码位置：`tj-promotion/src/main/resources/lua/receive_coupon.lua`

```lua
-- KEYS[1]: 优惠券信息Hash  KEYS[2]: 用户领券记录Hash
-- ARGV[1]: 用户ID
if(redis.call('exists', KEYS[1]) == 0) then
    return 1  -- 优惠券不存在
end
if(tonumber(redis.call('hget', KEYS[1], 'totalNum')) <= 0) then
    return 2  -- 库存不足
end
if(tonumber(redis.call('time')[1]) > tonumber(redis.call('hget', KEYS[1], 'issueEndTime'))) then
    return 3  -- 活动已结束
end
if(tonumber(redis.call('hget', KEYS[1], 'userLimit')) < redis.call('hincrby', KEYS[2], ARGV[1], 1)) then
    return 4  -- 超出限领数量
end
redis.call('hincrby', KEYS[1], "totalNum", "-1")  -- 扣减库存
return 0  -- 成功
```

**关键设计**：
- 4步校验（存在性→库存→时间→限领）在一个LUA脚本中**原子执行**
- `hincrby` 先自增再比较，即使并发也不会超领
- `redis.call('time')` 使用Redis服务器时间，避免各节点时钟不一致
- 返回值用数字编码表示不同错误类型

#### 6.3.2 兑换码兑换LUA脚本

> 源码位置：`tj-promotion/src/main/resources/lua/exchange_coupon.lua`

```lua
-- KEYS[1]: 兑换码Bitmap  KEYS[2]: 兑换码→优惠券映射ZSet
-- ARGV[1]: 序列号  ARGV[2]: 序列号+5000(范围上限)  ARGV[3]: 用户ID
if(redis.call('GETBIT', KEYS[1], ARGV[1]) == 1) then
    return "1"  -- 兑换码已被使用
end
local arr = redis.call('ZRANGEBYSCORE', KEYS[2], ARGV[1], ARGV[2], 'LIMIT', 0, 1);
if(#arr == 0) then
    return "2"  -- 兑换码不存在
end
local cid = arr[1]  -- 获取优惠券ID
-- ... 后续校验时间、限领数量

return cid  -- 返回优惠券ID
```

**关键设计**：
- **Bitmap** 存储兑换状态，1bit/码，43亿码仅需约512MB
- **ZSet** 存储序列号→优惠券ID的映射，通过 `ZRANGEBYSCORE` 范围查询定位优惠券
- GETBIT+SETBIT 原子标记，防止重复兑换

## 亮点七：优惠券叠加推荐算法 — 全排列 + 并行计算 + 最优解筛选

### 7.1 设计背景/问题

用户下单时可能有多张优惠券，需要推荐**最优的优惠券组合方案**：
- 不同优惠券可能限定不同课程范围
- 优惠券叠加使用时，前一张券的优惠会影响后一张券是否可用（门槛变化）
- 排列顺序不同，最终优惠金额可能不同
- 可用券数量多时，排列组合数量爆炸（N!级别），需要并行加速

### 7.2 解决方案

```
可用券筛选 → 全排列组合（回溯算法） → CompletableFuture并行计算每种方案
→ CountDownLatch超时保护 → 双维度最优解筛选
```

### 7.3 核心代码分析

#### 7.3.1 全排列工具类（回溯算法）

> 源码位置：`tj-promotion/src/main/java/com/tianji/promotion/utils/PermuteUtil.java`

```java
public class PermuteUtil {
    public static <T> List<List<T>> permute(List<T> input) {
        List<List<T>> res = new ArrayList<>();
        backtrack(input.size(), input, res, 0);
        return res;
    }
    private static <T> void backtrack(int n, List<T> input, List<List<T>> res, int first) {
        if (first == n) {
            res.add(new ArrayList<>(input));
        }
        for (int i = first; i < n; i++) {
            Collections.swap(input, first, i);   // 选择
            backtrack(n, input, res, first + 1);  // 递归
            Collections.swap(input, first, i);    // 撤销
        }
    }
}
```

#### 7.3.2 并行计算 + 超时保护

> 源码位置：`tj-promotion/src/main/java/com/tianji/promotion/service/impl/DiscountServiceImpl.java:40-93`

```java
public List<CouponDiscountDTO> findDiscountSolution(List<OrderCourseDTO> orderCourses) {
    // 1. 查询用户可用券 → 初筛（总价是否满足门槛）
    // 2. 细筛（每张券的适用课程范围）
    // 3. 全排列组合 + 单券方案
    List<List<Coupon>> solutions = PermuteUtil.permute(availableCoupons);
    for (Coupon c : availableCoupons) {
        solutions.add(List.of(c));  // 补充单券方案
    }
    // 4. 并行计算所有方案
    List<CouponDiscountDTO> list = Collections.synchronizedList(new ArrayList<>());
    CountDownLatch latch = new CountDownLatch(solutions.size());
    for (List<Coupon> solution : solutions) {
        CompletableFuture
            .supplyAsync(() -> calculateSolutionDiscount(..., solution),
                         discountSolutionExecutor)  // 专用线程池
            .thenAccept(dto -> {
                list.add(dto);
                latch.countDown();
            });
    }
    // 5. 超时保护：最多等待1秒
    latch.await(1, TimeUnit.SECONDS);
    // 6. 筛选最优解
    return findBestSolution(list);
}
```

#### 7.3.3 双维度最优解筛选

> 源码位置：`DiscountServiceImpl.java:112-145`

```java
private List<CouponDiscountDTO> findBestSolution(List<CouponDiscountDTO> list) {
    Map<String, CouponDiscountDTO> moreDiscountMap = new HashMap<>();  // 同券组合→最大优惠
    Map<Integer, CouponDiscountDTO> lessCouponMap = new HashMap<>();   // 同金额→最少用券
    for (CouponDiscountDTO solution : list) {
        // 维度1：相同用券组合中，保留优惠金额最大的
        // 维度2：相同优惠金额中，保留用券数量最少的
    }
    // 取两个维度的交集，按优惠金额降序返回
    return CollUtils.intersection(moreDiscountMap.values(), lessCouponMap.values())
        .stream().sorted(Comparator.comparingInt(
            CouponDiscountDTO::getDiscountAmount).reversed())
        .collect(Collectors.toList());
}
```

#### 7.3.4 专用线程池隔离

> 源码位置：`tj-promotion/src/main/java/com/tianji/promotion/config/PromotionConfig.java:33-47`

```java
@Bean
public Executor discountSolutionExecutor(){
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(12);
    executor.setMaxPoolSize(12);
    executor.setQueueCapacity(99999);

```

**设计要点**：
- 核心线程12个，利用多核CPU并行计算
- 队列容量99999，防止排列组合过多时丢弃任务
- 与兑换码生成线程池隔离（`generateExchangeCodeExecutor` 仅2核心5最大），互不影响

### 7.4 技术原理

| 维度 | 说明 |
|------|------|
| 全排列 | 回溯算法生成N!种排列，O(N*N!)时间复杂度 |
| 并行加速 | CompletableFuture + 12线程池，N!种方案并行计算 |
| 超时保护 | CountDownLatch.await(1s)，防止券太多导致计算时间过长 |
| 最优解 | 双维度交集：同组合最大优惠 ∩ 同金额最少用券 |
| 线程安全 | `Collections.synchronizedList` 保证并发写入安全 |

### 7.5 面试话术

> "优惠券叠加推荐的核心难点在于排列顺序会影响结果——前一张券优惠后，后续券的门槛可能不再满足。所以我们用回溯算法生成所有排列组合，再加上每张券单独使用的方案。由于排列数量是N!级别，我们使用CompletableFuture提交到12线程的专用线程池并行计算，同时用CountDownLatch设置1秒超时保护。最后从两个维度筛选最优解：相同组合取最大优惠，相同优惠取最少用券，两个维度取交集。线程池做了隔离设计，避免影响其他业务。"

## 亮点八：Redis合并写 + 异步持久化方案

### 8.1 设计背景/问题

项目中有多个场景存在高频写入需求：
- **视频播放进度**：前端每隔15秒上报一次，直接写DB压力巨大
- **点赞计数**：热门内容可能每秒数百次点赞/取消

直接写数据库会导致：大量UPDATE操作、行锁竞争、数据库CPU飙升

### 8.2 解决方案

统一模式：**写Redis缓存 → 合并/去重 → 定时任务/延迟任务批量持久化到MySQL**

### 8.3 方案一：视频播放进度（DelayQueue合并写）

> 源码位置：`tj-learning/src/main/java/com/tianji/learning/utils/LearningRecordDelayTaskHandler.java`

```
前端15秒上报 → 写Redis Hash缓存 → 投递DelayTask(20秒延迟)
                                          ↓
                    20秒后取出 → 比较moment值 → 相同则持久化 / 不同则丢弃
```

```java
public void addLearningRecordTask(LearningRecord record){
    // 1. 写Redis缓存（Hash结构：learning:record:{lessonId} → {sectionId: json}）
    writeRecordCache(record);
    // 2. 投递20秒延迟任务
    queue.add(new DelayTask<>(new RecordTaskData(record), Duration.ofSeconds(20)));
}

public void handleDelayTask(){
    while (begin) {
        DelayTask<RecordTaskData> task = queue.take();  // 阻塞等待到期任务
        LearningRecord record = readRecordCache(data.getLessonId(), data.getSectionId());
        // 比较moment：如果不一致说明用户还在看，丢弃旧数据等新的延迟任务
        if(!Objects.equals(data.getMoment(), record.getMoment())) {
            continue;  // 合并写：20秒内的多次上报只持久化最后一次
        }
        // moment一致，说明用户20秒没有新进度，持久化
        recordMapper.updateById(record);
    }
}
```

**效果**：假设用户持续观看1小时视频，前端上报240次（15秒/次），实际写DB仅约3次（暂停或视频切换时），**写入量降低约95%**。

## 亮点九：JVM DelayQueue 延迟去重机制

### 9.1 设计背景/问题

视频播放进度前端高频上报（每15秒），如果每次都持久化到数据库：
- 1个用户看1小时 = 240次DB写入
- 1000个用户同时在线 = 24万次/小时
- 大量无意义的中间状态写入

### 9.2 解决方案

> 源码位置：`tj-learning/src/main/java/com/tianji/learning/utils/DelayTask.java`

```java
public class DelayTask<D> implements Delayed {
    private D data;
    private long deadlineNanos;

    public DelayTask(D data, Duration delayTime) {
        this.data = data;
        this.deadlineNanos = System.nanoTime() + delayTime.toNanos();
    }

    @Override
    public long getDelay(TimeUnit unit) {
        return unit.convert(Math.max(0, deadlineNanos - System.nanoTime()), TimeUnit.NANOSECONDS);
    }

    @Override
    public int compareTo(Delayed o) {
        long l = getDelay(TimeUnit.NANOSECONDS) - o.getDelay(TimeUnit.NANOSECONDS);
        return l > 0 ? 1 : (l < 0 ? -1 : 0);
    }
}
```

### 9.3 延迟去重核心流程

```
                    写Redis缓存(moment=50)
T=0s   上报1  ──→  投递DelayTask(moment=50, 20秒后到期)
                    ↓
T=15s  上报2  ──→  更新Redis缓存(moment=65)
                    投递DelayTask(moment=65, 20秒后到期)
                    ↓
T=20s  Task1到期 → 读Redis缓存(moment=65) ≠ Task1的moment(50)
                    → 丢弃！用户还在看
                    ↓
T=30s  上报3  ──→  更新Redis缓存(moment=80)
                    投递DelayTask(moment=80, 20秒后到期)
                    ↓
T=35s  Task2到期 → 读Redis缓存(moment=80) ≠ Task2的moment(65)
                    → 丢弃！
                    ↓
T=50s  Task3到期 → 读Redis缓存(moment=80) == Task3的moment(80)
                    → 持久化到MySQL！用户已暂停20秒
```

### 9.4 生命周期管理

```java
@PostConstruct
public void init(){
    CompletableFuture.runAsync(this::handleDelayTask);  // 应用启动时开启消费线程
}

@PreDestroy
public void destroy(){
    begin = false;  // volatile变量，优雅关闭
}
```

### 9.5 面试话术

> "我使用JVM的DelayQueue实现了视频播放进度的延迟去重。每次前端上报进度时，先更新Redis缓存中的最新moment值，再投递一个携带当前moment的20秒延迟任务。任务到期后比较自身moment和Redis中的最新moment，如果不一致说明中间有新的上报，直接丢弃；一致则说明用户已暂停20秒以上，执行持久化。这样多次上报只有最后一次会真正写库。应用启动时用@PostConstruct开启消费线程，@PreDestroy通过volatile标志优雅关闭。"

## 亮点十：CompletableFuture + 线程池隔离的并发编程实践

### 10.1 项目中的并发编程应用场景

| 场景 | 位置 | 技术 | 目的 |

| 优惠券方案并行计算 | DiscountServiceImpl | CompletableFuture + CountDownLatch | N!种方案并行计算 |

### 10.2 线程池隔离设计

> 源码位置：`tj-promotion/src/main/java/com/tianji/promotion/config/PromotionConfig.java`

项目配置了两个独立线程池，实现**资源隔离**：

```java

```

### 10.3 设计要点

| 设计决策 | 兑换码线程池 | 方案计算线程池 | 原因 |
|----------|------------|--------------|------|
| 核心线程 | 2 | 12 | IO密集vs CPU密集 |
| 最大线程 | 5 | 12 | 允许弹性vs固定避免切换 |
| 队列容量 | 200 | 99999 | 任务量可控vs排列爆炸 |
| 拒绝策略 | CallerRunsPolicy | AbortPolicy | 降级执行vs快速失败 |
| 线程名前缀 | exchange-code-handler- | discount-solution-calculator- | 便于排查 |

### 10.4 CountDownLatch超时保护模式

```java
// 1. 创建闭锁
CountDownLatch latch = new CountDownLatch(solutions.size());

// 2. 异步提交所有任务
for (List<Coupon> solution : solutions) {
    CompletableFuture.supplyAsync(() -> calculate(solution), executor)
        .thenAccept(dto -> {
            list.add(dto);
            latch.countDown();  // 每个任务完成时计数-1
        });
}

// 3. 超时等待（最多1秒）
latch.await(1, TimeUnit.SECONDS);
// 即使部分任务未完成，也能返回已计算的结果
```

**为什么不用 CompletableFuture.allOf？**
- `allOf` 没有原生超时支持（需要额外包装）
- CountDownLatch.await(timeout) 更直观，超时后直接返回已有结果

### 10.5 面试话术

> "项目中我在多个场景使用了并发编程。最典型的是优惠券叠加推荐：排列组合可能产生N!种方案，我使用CompletableFuture.supplyAsync提交到12线程的专用线程池并行计算，同时用CountDownLatch设置1秒超时保护，即使部分方案没算完也能返回已有结果。线程池做了隔离设计——方案计算池是CPU密集型，12核心固定大小；兑换码生成池是IO密集型，2核心5最大可弹性扩容。拒绝策略也不同：计算池用AbortPolicy快速失败，生成池用CallerRunsPolicy降级由调用线程执行，保证兑换码不丢失。"

## 总结：技术亮点面试回答框架

面试中回答技术方案时，建议按以下框架组织：

1. **背景**：遇到了什么问题？（高并发/数据量大/性能瓶颈...）
2. **方案**：如何解决？（一句话概括核心思路）
3. **实现**：关键技术细节（数据结构、算法、中间件选型）
4. **效果**：带来了什么提升？（量化数据最佳）
5. **权衡**：为什么选择这个方案而不是其他？（体现思考深度）
