## 亮点一：基于注解 + AOP + 策略模式的分布式锁组件

### 1.1 设计背景与问题

在微服务架构中，分布式锁是高并发场景下保证数据一致性的核心手段。传统的分布式锁使用方式存在以下痛点：

- **代码侵入性强**：每次使用都需要手动编写 `tryLock / unlock` 逻辑，存在大量模板代码。
- **锁类型不可配置**：切换可重入锁、公平锁、读写锁需要修改业务代码。
- **获锁失败策略僵化**：获取锁失败后是重试、跳过、还是抛异常，缺少灵活的控制手段。
- **锁粒度固定**：锁的 key 通常是硬编码字符串，无法根据运行时参数动态生成。

### 1.2 解决方案

项目中设计了一套**声明式分布式锁组件**，核心思路为 **自定义注解 + AOP环绕通知 + 策略模式 + 简单工厂 + SPEL表达式动态锁名**。

整体架构如下：

```
@Lock 注解（声明式使用）
    |
    v
LockAspect（AOP切面，环绕通知）
    |
    +---> LockType 枚举（锁类型工厂：可重入锁 / 公平锁 / 读锁 / 写锁）
    |
    +---> LockStrategy 枚举（加锁策略：快速失败 / 快速跳过 / 重试超时失败 / 重试超时跳过 / 永久等待）
    |
    +---> SPEL 表达式解析器（动态锁名）
```

项目中实际存在两套分布式锁实现：
- **tj-common 中的通用方案**：`@Lock` + `LockAspect` + `LockType` + `LockStrategy`，适用于所有微服务。
- **tj-promotion 中的进阶方案**：`@MyLock` + `MyLockAspect` + `MyLockFactory` + `MyLockType` + `MyLockStrategy`，使用简单工厂模式（`EnumMap + Function`）进一步解耦。

### 1.3 核心代码分析

#### 1.3.1 @Lock 注解定义

> 源码位置：`tj-common/src/main/java/com/tianji/common/autoconfigure/redisson/annotations/Lock.java`

```java
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
public @interface Lock {
    /** 加锁key的表达式，支持SPEL表达式 */
    String name();
    /** 阻塞超时时长 */
    long waitTime() default 1;
    /** 锁自动释放时长，默认-1（30秒 + watchDog模式） */
    long leaseTime() default -1;
    /** 时间单位 */
    TimeUnit timeUnit() default TimeUnit.SECONDS;
    /** 方法结束是否自动释放锁 */
    boolean autoUnlock() default true;
    /** 锁的类型 */
    LockType lockType() default LockType.DEFAULT;
    /** 锁策略 */
    LockStrategy lockStrategy() default LockStrategy.FAIL_AFTER_RETRY_TIMEOUT;
}
```

**设计要点**：
- `leaseTime = -1` 时启用 Redisson 的 WatchDog 机制，自动续期，避免业务未执行完锁就过期。
- `autoUnlock = false` 可以让锁在方法结束后不释放，等待自然过期，适用于"幂等性防重"场景。
- 将锁类型和获锁策略作为注解属性暴露，使用时只需声明，无需修改代码即可切换。

#### 1.3.2 LockType -- 锁类型枚举（工厂方法模式）

> 源码位置：`tj-common/src/main/java/com/tianji/common/autoconfigure/redisson/enums/LockType.java`

```java
public enum LockType {
    DEFAULT(){
        @Override
        public RLock getLock(RedissonClient redissonClient, String name) {
            return redissonClient.getLock(name);   // 可重入锁
        }
    },
    FAIR_LOCK(){
        @Override
        public RLock getLock(RedissonClient redissonClient, String name) {
            return redissonClient.getFairLock(name);   // 公平锁
        }
    },
    READ_LOCK(){
        @Override
        public RLock getLock(RedissonClient redissonClient, String name) {
            return redissonClient.getReadWriteLock(name).readLock();   // 读锁
        }
    },
    WRITE_LOCK(){
        @Override
        public RLock getLock(RedissonClient redissonClient, String name) {
            return redissonClient.getReadWriteLock(name).writeLock();  // 写锁
        }
    };
    public abstract RLock getLock(RedissonClient redissonClient, String name);
}
```

**设计要点**：每个枚举值实现抽象方法 `getLock()`，充当**工厂方法**角色。调用方只需传入枚举值即可获取对应类型的锁对象，完全屏蔽了 Redisson API 的差异。

#### 1.3.3 LockStrategy -- 加锁策略枚举（策略模式）

> 源码位置：`tj-common/src/main/java/com/tianji/common/autoconfigure/redisson/enums/LockStrategy.java`

```java
public enum LockStrategy {
    SKIP_FAST() {           // 不重试，直接返回false
        @Override
        public boolean tryLock(RLock lock, Lock prop) throws InterruptedException {
            return lock.tryLock(0, prop.leaseTime(), prop.timeUnit());
        }
    },
    FAIL_FAST() {           // 不重试，直接抛异常
        @Override
        public boolean tryLock(RLock lock, Lock prop) throws InterruptedException {
            boolean success = lock.tryLock(0, prop.leaseTime(), prop.timeUnit());
            if (!success) {
                throw new BizIllegalException("请求太频繁");
            }
            return true;
        }
    },
    SKIP_AFTER_RETRY_TIMEOUT() {  // 重试到超时，返回false
        @Override
        public boolean tryLock(RLock lock, Lock prop) throws InterruptedException {
            return lock.tryLock(prop.waitTime(), prop.leaseTime(), prop.timeUnit());
        }
    },
    FAIL_AFTER_RETRY_TIMEOUT() {  // 重试到超时，抛异常（默认策略）
        @Override
        public boolean tryLock(RLock lock, Lock prop) throws InterruptedException {
            boolean success = lock.tryLock(prop.waitTime(), prop.leaseTime(), prop.timeUnit());
            if (!success) {
                throw new BizIllegalException("请求超时");
            }
            return true;
        }
    },
    KEEP_RETRY() {          // 永久等待，直到成功
        @Override
        public boolean tryLock(RLock lock, Lock prop) throws InterruptedException {
            lock.lock(prop.leaseTime(), prop.timeUnit());
            return true;
        }
    };
    public abstract boolean tryLock(RLock lock, Lock prop) throws InterruptedException;
}
```

**设计要点**：5种策略覆盖了所有业务场景：
| 策略 | 重试 | 失败处理 | 适用场景 |
|---|---|---|---|
| `SKIP_FAST` | 不重试 | 返回null | 防重提交 |
| `FAIL_FAST` | 不重试 | 抛异常 | 秒杀下单 |
| `SKIP_AFTER_RETRY_TIMEOUT` | 重试 | 返回null | 异步任务 |
| `FAIL_AFTER_RETRY_TIMEOUT` | 重试 | 抛异常 | 通用业务（默认） |
| `KEEP_RETRY` | 永久等待 | -- | 必须执行的任务 |

#### 1.3.4 LockAspect -- AOP切面（核心调度器）

> 源码位置：`tj-common/src/main/java/com/tianji/common/autoconfigure/redisson/aspect/LockAspect.java`（第37-61行）

```java
@Around("@annotation(properties)")
public Object handleLock(ProceedingJoinPoint pjp, Lock properties) throws Throwable {
    if (!properties.autoUnlock() && properties.leaseTime() <= 0) {
        throw new BizIllegalException("leaseTime不能为空");
    }
    // 1.基于SPEL表达式解析锁的 name
    String name = getLockName(properties.name(), pjp);
    // 2.得到锁对象（工厂方法）
    RLock rLock = properties.lockType().getLock(redissonClient, name);
    // 3.尝试获取锁（策略模式）
    boolean success = properties.lockStrategy().tryLock(rLock, properties);
    if (!success) {
        return null;
    }
    try {
        // 4.执行被代理方法
        return pjp.proceed();
    } finally {
        // 5.释放锁
        if (properties.autoUnlock()) {
            rLock.unlock();
        }
    }
}
```

**核心流程**：切面中不包含任何具体的加锁逻辑和锁类型判断，所有差异化行为全部委托给枚举实现。切面本身只做流程编排，符合"开闭原则"。

#### 1.3.5 SPEL动态锁名解析

> 源码位置：`tj-common/src/main/java/com/tianji/common/autoconfigure/redisson/aspect/LockAspect.java`（第66-101行）

```java
private static final Pattern pattern = Pattern.compile("\\#\\{([^\\}]*)\\}");

private String getLockName(String name, ProceedingJoinPoint pjp) {
    if (StringUtils.isBlank(name) || !name.contains("#")) {
        return name;    // 没有SPEL表达式，直接返回
    }
    // 构建SPEL上下文，将方法参数注入
    EvaluationContext context = new MethodBasedEvaluationContext(
            TypedValue.NULL, resolveMethod(pjp), pjp.getArgs(), parameterNameDiscoverer);
    ExpressionParser parser = new SpelExpressionParser();
    Matcher matcher = pattern.matcher(name);
    while (matcher.find()) {
        String tmp = matcher.group();
        String group = matcher.group(1);
        Expression expression = parser.parseExpression(
            group.charAt(0) == 'T' ? group : "#" + group);
        Object value = expression.getValue(context);
        name = name.replace(tmp, ObjectUtils.nullSafeToString(value));
    }
    return name;
}
```

**使用示例**：
```java
@Lock(name = "lock:coupon:#{userId}")
public void bindCoupon(Long userId) { ... }
```

此时如果 `userId = 100`，实际锁的 key 为 `lock:coupon:100`，实现了**用户级别的细粒度锁**。

#### 1.3.6 tj-promotion 中的进阶方案 -- 简单工厂

> 源码位置：`tj-promotion/src/main/java/com/tianji/promotion/utils/MyLockFactory.java`

```java
@Component
public class MyLockFactory {
    private final Map<MyLockType, Function<String, RLock>> lockHandlers;

    public MyLockFactory(RedissonClient redissonClient) {
        this.lockHandlers = new EnumMap<>(MyLockType.class);
        this.lockHandlers.put(RE_ENTRANT_LOCK, redissonClient::getLock);
        this.lockHandlers.put(FAIR_LOCK, redissonClient::getFairLock);
        this.lockHandlers.put(READ_LOCK, name -> redissonClient.getReadWriteLock(name).readLock());
        this.lockHandlers.put(WRITE_LOCK, name -> redissonClient.getReadWriteLock(name).writeLock());
    }

    public RLock getLock(MyLockType lockType, String name){
        return lockHandlers.get(lockType).apply(name);
    }
}
```

**与 tj-common 方案的对比**：
| 维度 | tj-common 方案（LockType枚举） | tj-promotion 方案（MyLockFactory） |
|---|---|---|
| 锁类型选择 | 枚举抽象方法，每个枚举实现 `getLock()` | EnumMap + Function，工厂集中管理 |
| 扩展性 | 新增锁类型需修改枚举 | 新增锁类型只需在工厂中 put 新映射 |
| 设计模式 | 工厂方法模式 | 简单工厂 + 策略（Function） |
| Spring集成 | 枚举不受Spring管理 | @Component 交由Spring容器管理 |

### 1.4 技术原理

1. **AOP 环绕通知**：利用 `@Around` 在方法执行前后插入加锁/解锁逻辑，实现声明式编程。
2. **策略模式**：将加锁策略封装到枚举的抽象方法中，每个枚举值代表一种策略，通过注解参数动态切换。
3. **工厂方法/简单工厂**：`LockType` 枚举的每个值充当工厂方法创建不同锁对象；`MyLockFactory` 则用 `EnumMap<LockType, Function>` 实现简单工厂。
4. **SPEL 表达式引擎**：通过正则 `#{expr}` 提取表达式，借助 `MethodBasedEvaluationContext` 将方法参数注入上下文，实现运行时动态解析。
5. **Redisson WatchDog**：当 `leaseTime = -1` 时，Redisson 自动启用看门狗机制，默认每 10 秒续期一次，直到业务执行完毕。

### 1.5 面试话术建议

> **面试官问**：你的项目中分布式锁是怎么用的？
>
> **回答**：我在项目中封装了一个基于 Redisson 的声明式分布式锁组件。核心设计是**自定义注解 + AOP + 策略模式**。
>
> 首先定义了一个 `@Lock` 注解，包含锁名称、等待时长、过期时间、锁类型、加锁策略等属性。然后通过 AOP 环绕通知拦截所有标注了 `@Lock` 的方法。
>
> 锁类型的切换用了**工厂方法模式**，我用枚举来实现，每个枚举值（可重入锁、公平锁、读锁、写锁）都实现了 `getLock()` 方法，调用方只需在注解中声明类型即可，不需要关心底层 Redisson API。
>
> 加锁策略用了**策略模式**，同样用枚举实现，提供了 5 种策略：快速失败、快速跳过、重试后失败、重试后跳过、永久等待。不同业务场景选择不同策略，比如秒杀用快速失败，普通业务用重试后失败。
>
> 锁名称支持 **SPEL 表达式**，比如 `lock:user:#{userId}`，切面会在运行时解析方法参数并替换，实现用户级细粒度加锁。
>
> 另外在优惠券模块，我还做了进一步优化，用 `EnumMap + Function` 实现简单工厂，将锁的创建逻辑集中管理，扩展性更好。

## 亮点二：视频播放进度记录系统（DelayQueue + Redis缓存 + 合并写策略）

### 2.1 设计背景与问题

在线教育平台中，视频播放进度是核心功能。前端每隔约 **15秒** 就会提交一次播放进度（`moment`），要求：

- **精确到秒**的播放进度记录，用户下次打开能从上次位置继续播放。
- 判断是否完成视频学习（播放超过总时长一半则标记为已完成）。
- 每个用户同时可能在学习多门课程。

**性能挑战**：假设平台有 5 万用户同时在线学习，每人每 15 秒提交一次进度，则系统每秒需要处理约 **3333 次数据库写操作**，这对 MySQL 来说是不可接受的压力。

### 2.2 解决方案

项目采用了 **Redis缓存 + JVM DelayQueue延迟队列 + 合并写** 的三层架构：

```
前端提交播放进度（每15秒）
    |
    v
LearningRecordServiceImpl.handleVideoRecord()
    |
    +---> [1] Redis Hash 缓存记录（覆盖写）
    |        Key = learning:record:{lessonId}
    |        Field = sectionId
    |        Value = {id, moment, finished} (JSON)
    |
    +---> [2] JVM DelayQueue 提交延迟任务（20秒后执行）
    |
    v
LearningRecordDelayTaskHandler.handleDelayTask()（后台线程消费）
    |
    +---> [3] 从Redis读取最新进度
    +---> [4] 对比moment，若与任务中的一致 => 写入MySQL
    |        （若不一致 => 说明用户仍在提交新进度，丢弃旧任务）
```

### 2.3 核心代码分析

#### 2.3.1 泛型延迟任务封装

> 源码位置：`tj-learning/src/main/java/com/tianji/learning/utils/DelayTask.java`

```java
@Data
public class DelayTask<D> implements Delayed {
    private D data;
    private long deadlineNanos;

    public DelayTask(D data, Duration delayTime) {
        this.data = data;
        this.deadlineNanos = System.nanoTime() + delayTime.toNanos();
    }

    @Override
    public long getDelay(TimeUnit unit) {
        return unit.convert(Math.max(0, deadlineNanos - System.nanoTime()), TimeUnit.NANOSECONDS);
    }

    @Override
    public int compareTo(Delayed o) {
        long l = getDelay(TimeUnit.NANOSECONDS) - o.getDelay(TimeUnit.NANOSECONDS);
        if(l > 0) return 1;
        else if(l < 0) return -1;
        else return 0;
    }
}
```

**设计要点**：使用纳秒精度 `System.nanoTime()` 而非 `System.currentTimeMillis()`，避免系统时钟调整带来的影响。泛型设计使其可复用于任意业务场景。

#### 2.3.2 延迟任务处理器（核心）

> 源码位置：`tj-learning/src/main/java/com/tianji/learning/utils/LearningRecordDelayTaskHandler.java`

**初始化与生命周期管理**（第36-44行）：
```java
private final DelayQueue<DelayTask<RecordTaskData>> queue = new DelayQueue<>();
private static volatile boolean begin = true;

@PostConstruct
public void init(){
    CompletableFuture.runAsync(this::handleDelayTask);  // 启动后台消费线程
}

@PreDestroy
public void destroy(){
    begin = false;  // 优雅停机
}
```

**后台消费线程**（第46-77行）：
```java
public void handleDelayTask(){
    while (begin) {
        try {
            // 1.获取到期的延迟任务（阻塞等待）
            DelayTask<RecordTaskData> task = queue.take();
            RecordTaskData data = task.getData();
            // 2.查询Redis缓存中的最新进度
            LearningRecord record = readRecordCache(data.getLessonId(), data.getSectionId());
            if (record == null) {
                continue;
            }
            // 3.比较moment值
            if(!Objects.equals(data.getMoment(), record.getMoment())) {
                // 不一致，说明用户还在持续提交播放进度，放弃旧数据
                continue;
            }
            // 4.一致，说明用户已停止播放或暂停，持久化到数据库
            record.setFinished(null);
            recordMapper.updateById(record);
            // 4.2.更新课表最近学习信息
            LearningLesson lesson = new LearningLesson();
            lesson.setId(data.getLessonId());
            lesson.setLatestSectionId(data.getSectionId());
            lesson.setLatestLearnTime(LocalDateTime.now());
            lessonService.updateById(lesson);
        } catch (Exception e) {
            log.error("处理延迟任务发生异常", e);
        }
    }
}
```

**提交延迟任务与缓存操作**（第79-121行）：

```java
public void addLearningRecordTask(LearningRecord record){

    writeRecordCache(record);
```

#### 2.3.3 业务层调用逻辑

> 源码位置：`tj-learning/src/main/java/com/tianji/learning/service/impl/LearningRecordServiceImpl.java`（第110-170行）

```java
private boolean handleVideoRecord(Long userId, LearningRecordFormDTO recordDTO) {
    // 1.查询旧的学习记录（优先从Redis缓存查询）
    LearningRecord old = queryOldRecord(recordDTO.getLessonId(), recordDTO.getSectionId());
    if (old == null) {
        // 2.不存在，新增记录到数据库
        LearningRecord record = BeanUtils.copyBean(recordDTO, LearningRecord.class);
        record.setUserId(userId);
        save(record);
        return false;
    }
    // 3.存在，判断是否是第一次完成（播放进度超过总时长一半）
    boolean finished = !old.getFinished() && recordDTO.getMoment() * 2 >= recordDTO.getDuration();
    if(!finished){
        // 4.未完成 => 仅更新缓存 + 提交延迟任务（不写DB）
        LearningRecord record = new LearningRecord();
        record.setLessonId(recordDTO.getLessonId());
        record.setSectionId(recordDTO.getSectionId());
        record.setMoment(recordDTO.getMoment());
        record.setId(old.getId());
        record.setFinished(old.getFinished());
        taskHandler.addLearningRecordTask(record);  // 写Redis + 延迟队列
        return false;
    }
    // 5.第一次完成 => 立即写入数据库
    lambdaUpdate()
        .set(LearningRecord::getMoment, recordDTO.getMoment())
        .set(LearningRecord::getFinished, true)
        .set(LearningRecord::getFinishTime, recordDTO.getCommitTime())
        .eq(LearningRecord::getId, old.getId())
        .update();
    taskHandler.cleanRecordCache(recordDTO.getLessonId(), recordDTO.getSectionId());
    return true;
}
```

### 2.4 技术原理

**合并写（Write Merging）思想**：

前端每15秒提交一次进度，但我们不需要每次都写入数据库。核心思路是：
1. 将最新进度写入 Redis（覆盖式更新，只保留最新值）。
2. 同时提交一个 20秒延迟任务到 JVM 的 `DelayQueue`。
3. 延迟任务到期后，取出 Redis 中的最新进度与任务中的进度对比：
   - **相同** => 说明用户在这 20 秒内没有新的提交，进度已稳定，执行持久化。
   - **不同** => 说明用户还在持续播放并提交新进度，丢弃本次任务（后续任务会负责持久化）。

**效果量化**：假设用户持续播放 5 分钟，前端提交约 20 次进度，但实际只会触发 1 次数据库写入（最后一次稳定后），写入量降低约 **95%**。

**Redis 缓存设计**：
- 使用 **Hash 结构**：`Key = learning:record:{lessonId}`，`Field = sectionId`。
- 一个课程的所有小节进度集中在一个 Key 下，查询效率高。
- TTL 设为 1 分钟，自动清理不活跃数据。

### 2.5 面试话术建议

> **面试官问**：视频播放进度是怎么记录的？高频提交怎么处理的？
>
> **回答**：这是我设计的一个高频写入场景的优化方案。前端每15秒提交一次播放进度，如果直接写数据库，5万人在线每秒就有3000+次写操作，MySQL扛不住。
>
> 我采用了 **Redis缓存 + JVM DelayQueue + 合并写** 的方案：
>
> 1. 每次收到进度提交，先写入 Redis Hash（覆盖式更新），同时向 JVM 的 DelayQueue 投递一个 20 秒延迟任务。
> 2. 后台有一个 `@PostConstruct` 启动的消费线程，通过 `queue.take()` 阻塞等待到期任务。
> 3. 任务到期后，取 Redis 中最新的 moment 与任务中的 moment 比较：如果相同，说明用户已停止播放，执行一次 DB 更新；如果不同，说明用户还在播放，直接丢弃旧任务。
>
> 这样用户连续播放 5 分钟提交 20 次进度，实际只写入 DB 1 次，写入量降低 95%。
>
> 另外有一个特殊情况：如果用户播放进度达到总时长的一半，标记为"已完成"，这个操作需要**立即写入数据库**，不走延迟队列，保证完课状态的实时性。

## 亮点三：点赞系统设计（Redis Set/ZSet + 定时任务 + Pipeline批量查询）

### 3.1 设计背景与问题

点赞是互联网应用中极高频的操作，存在以下挑战：

- **高频读写**：用户可以对问答、笔记等多种业务进行点赞/取消，QPS 很高。
- **防重**：同一用户不能对同一内容重复点赞。
- **实时统计**：前端需要实时显示点赞总数和当前用户的点赞状态。
- **批量查询**：一个列表页可能展示 10-20 条内容，需要批量查询用户是否已点赞。
- **持久化需求**：点赞数据最终需要同步到 MySQL，驱动业务排序等功能。

### 3.2 解决方案

项目采用 **Redis 双数据结构 + 定时任务异步持久化 + MQ 解耦 + Pipeline 批量查询** 的方案：

```
用户点赞/取消
    |
    v
Redis Set（记录每个业务的点赞用户集合）
    Key: likes:set:biz:{bizId}
    Members: userId1, userId2, ...
    |
    +---> SCARD 统计点赞总数
    |
    v
Redis ZSet（缓存点赞数变更的缓冲区）
    Key: likes:times:type:{bizType}
    Member: bizId
    Score: likedTimes
    |
    v
定时任务（每20秒）
    |
    +---> ZPOPMIN 批量取出变更记录
    +---> 发送MQ消息到对应业务服务
    v
业务服务（tj-learning）
    +---> MQ消费者更新MySQL中的点赞数字段
```

### 3.3 核心代码分析

**设计要点**：
- **Redis Set 天然去重**：`SADD` 对已存在的元素返回 0，天然防止重复点赞，无需额外判断。
- **原子性**：`SADD` 和 `SREM` 是 Redis 原子操作，高并发下不会出现数据不一致。
- **ZSet 作为变更缓冲区**：每次点赞/取消后，将最新的点赞总数写入 ZSet。相同 bizId 的多次操作会自动覆盖（ZSet 中 member 唯一），所以 ZSet 中始终保存的是**最新的点赞总数**。

#### 3.3.3 Pipeline 批量查询点赞状态

> 源码位置：`tj-remark/src/main/java/com/tianji/remark/service/impl/LikedRecordServiceRedisImpl.java`（第67-84行）

```java
@Override
public Set<Long> isBizLiked(List<Long> bizIds) {
    Long userId = UserContext.getUser();
    // 1.使用Pipeline批量查询，一次网络往返完成所有查询
    List<Object> objects = redisTemplate.executePipelined((RedisCallback<Object>) connection -> {
        StringRedisConnection src = (StringRedisConnection) connection;
        for (Long bizId : bizIds) {
            String key = RedisConstants.LIKES_BIZ_KEY_PREFIX + bizId;
            src.sIsMember(key, userId.toString());
        }
        return null;
    });
    // 2.过滤出已点赞的bizId
    return IntStream.range(0, objects.size())
            .filter(i -> (boolean) objects.get(i))
            .mapToObj(bizIds::get)
            .collect(Collectors.toSet());
}
```

**设计要点**：列表页需要展示多条内容的点赞状态，如果逐个查询 Redis，N 条内容需要 N 次网络往返。使用 **Pipeline** 将所有 `SISMEMBER` 命令打包在一次网络请求中发送，性能提升显著。

```java
@Override
public void readLikedTimesAndSendMessage(String bizType, int maxBizSize) {
    String key = RedisConstants.LIKES_TIMES_KEY_PREFIX + bizType;
    // 1.ZPOPMIN：取出score最小的N条记录并从ZSet中移除
    Set<ZSetOperations.TypedTuple<String>> tuples =
        redisTemplate.opsForZSet().popMin(key, maxBizSize);
    if (CollUtils.isEmpty(tuples)) {
        return;
    }
    // 2.转换为DTO
    List<LikedTimesDTO> list = new ArrayList<>(tuples.size());
    for (ZSetOperations.TypedTuple<String> tuple : tuples) {
        String bizId = tuple.getValue();
        Double likedTimes = tuple.getScore();
        if (bizId == null || likedTimes == null) continue;
        list.add(LikedTimesDTO.of(Long.valueOf(bizId), likedTimes.intValue()));
    }
    // 3.发送MQ消息到对应业务服务
    mqHelper.send(
        LIKE_RECORD_EXCHANGE,
        StringUtils.format(LIKED_TIMES_KEY_TEMPLATE, bizType),
        list);
}
```

**设计要点**：
- `ZPOPMIN` 是 Redis 原子操作，取出并删除，避免重复处理。
- 每次最多取 30 条，防止单次处理量过大。
- 通过 MQ 解耦：点赞服务不直接操作业务表，而是发消息由业务服务自行更新，实现服务间松耦合。

### 3.4 技术原理

| 组件 | Redis 数据结构 | 用途 | 操作 |
|---|---|---|---|
| 点赞记录 | **Set** (`likes:set:biz:{bizId}`) | 存储每个业务的点赞用户集合 | SADD / SREM / SISMEMBER / SCARD |
| 变更缓冲区 | **ZSet** (`likes:times:type:{bizType}`) | 缓存有变更的业务及其最新点赞数 | ZADD（覆盖） / ZPOPMIN（消费） |

整个流程形成了一个 **"写缓冲 + 异步刷盘"** 的模式：
1. 点赞时实时写 Redis Set（毫秒级响应），同时更新 ZSet 缓冲。
2. 定时任务每 20 秒从 ZSet 中取出变更数据，通过 MQ 异步同步到 MySQL。
3. 即使 MySQL 短暂不可用，Redis 中的数据也不会丢失。

### 3.5 面试话术建议

> **面试官问**：你的点赞功能是怎么实现的？
>
> **回答**：我设计了一个基于 Redis 双数据结构的点赞系统。
>
> 点赞记录用 **Redis Set** 存储，Key 是业务ID，Members 是用户ID。Set 天然去重，`SADD` 返回 0 就说明已经点过了，完美解决防重问题。`SCARD` 可以直接获取点赞总数，`SISMEMBER` 判断是否已点赞。
>
> 批量查询点赞状态时，用了 **Redis Pipeline**，把多个 `SISMEMBER` 打包成一次网络请求，避免了 N 次网络往返。

## 亮点四：积分排行榜系统（BitMap签到 + ZSet实时榜 + MySQL动态分表）

### 4.1 设计背景与问题

积分排行榜系统涉及三个子系统：

1. **签到系统**：用户每日签到获取积分，连续签到有额外奖励。
2. **实时排行榜**：当月积分实时排名，用户可以看到自己的排名和榜单。
3. **历史排行榜**：每月赛季结束后，将排行榜数据持久化到 MySQL，供历史查询。

**挑战**：
- 签到数据量大但查询简单（只需判断某天是否签到），用 MySQL 存储浪费空间。
- 排行榜需要实时排名查询，MySQL `ORDER BY + LIMIT` 在大数据量下性能差。
- 历史排行榜每月一份，若放同一张表会越来越大，需要分表。

### 4.2 解决方案

```
签到 => Redis BitMap（每用户每月1个Key，32 bit 存31天签到记录）
    |
    +---> MQ消息 => 积分服务 => Redis ZSet（当月实时排行榜）
    |
    v
定时任务（月末）=> XxlJob分布式调度
    |
    +---> [1] 创建新的MySQL分表：points_board_{seasonId}
    +---> [2] 从Redis ZSet分页读取 => 批量写入MySQL分表
    +---> [3] 清理Redis中上月数据
```

```java
public SignResultVO addSignRecords() {
```

**连续签到天数计算**（第105-124行）：
```java
private int countSignDays(String key, int len) {
    // 1.BITFIELD key GET u{len} 0 —— 获取从第0位开始的len位无符号数
    List<Long> result = redisTemplate.opsForValue()
            .bitField(key, BitFieldSubCommands.create().get(
                    BitFieldSubCommands.BitFieldType.unsigned(len)).valueAt(0));
    if (CollUtils.isEmpty(result)) {
        return 0;
    }
    int num = result.get(0).intValue();
    int count = 0;
    // 2.从最低位开始，逐位检查。遇到0就终止（连续签到中断）
    while ((num & 1) == 1) {
        count++;
        num >>>= 1;  // 无符号右移
    }
    return count;
}
```

**查询本月签到记录**（第76-103行）：
```java
@Override
public Byte[] querySignRecords() {
    Long userId = UserContext.getUser();
    LocalDate now = LocalDate.now();
    int dayOfMonth = now.getDayOfMonth();
    String key = RedisConstants.SIGN_RECORD_KEY_PREFIX + userId
            + now.format(DateUtils.SIGN_DATE_SUFFIX_FORMATTER);
    // BITFIELD key GET u{dayOfMonth} 0
    List<Long> result = redisTemplate.opsForValue()
            .bitField(key, BitFieldSubCommands.create().get(
                    BitFieldSubCommands.BitFieldType.unsigned(dayOfMonth)).valueAt(0));
    if (CollUtils.isEmpty(result)) {
        return new Byte[0];
    }
    int num = result.get(0).intValue();
    Byte[] arr = new Byte[dayOfMonth];
    int pos = dayOfMonth - 1;
    while (pos >= 0){
        arr[pos--] = (byte)(num & 1);  // 逐位提取
        num >>>= 1;
    }
    return arr;
}
```

**设计要点**：
- 一个用户一个月的签到记录只需 **4 字节**（31 bit），100 万用户只需约 4MB 内存。
- `SETBIT` 的返回值天然判断是否重复签到。
- `BITFIELD GET` 一次性取出所有位数据，再用位运算解析。

```java
    // XxlJob 分片参数，支持多节点并行处理
    int index = XxlJobHelper.getShardIndex();
    int total = XxlJobHelper.getShardTotal();
    int pageNo = index + 1;
    int pageSize = 10;
    while (true) {
        // 从Redis ZSet分页读取
        List<PointsBoard> boardList = pointsBoardService.queryCurrentBoardList(key, pageNo, pageSize);
        if (CollUtils.isEmpty(boardList)) break;
        // 把排名写入id字段（历史表中id就是排名）
        boardList.forEach(b -> {
            b.setId(b.getRank().longValue());
            b.setRank(null);
        });
        // 批量写入MySQL分表
        pointsBoardService.saveBatch(boardList);
        pageNo += total;   // 分片跳页
    }
    TableInfoContext.remove();
}
```

### 4.4 技术原理

| 场景 | 技术方案 | 数据结构/存储 | 优势 |
|---|---|---|---|
| 签到 | Redis BitMap | `SETBIT`/`GETBIT`/`BITFIELD` | 一个用户一个月仅需4字节 |
| 连续签到 | 位运算 | `num & 1` 逐位检测 | O(n) 纯内存运算，n最大31 |
| 实时排行榜 | Redis ZSet | `ZINCRBY`/`ZREVRANGE`/`ZREVRANK` | 增删改查均为 O(log N) |
| 历史排行榜 | MySQL 动态分表 | `points_board_{seasonId}` | 每月独立分表，避免数据膨胀 |
| 动态表名 | ThreadLocal + MyBatis-Plus 拦截器 | `DynamicTableNameInnerInterceptor` | 对业务代码透明 |
| 持久化调度 | XxlJob 分片广播 | `ShardIndex`/`ShardTotal` | 多节点并行，提升持久化速度 |

**分片广播原理**：假设有 3 个执行器节点，XxlJob 分片广播时：
- 节点0：处理第 1、4、7、10... 页
- 节点1：处理第 2、5、8、11... 页
- 节点2：处理第 3、6、9、12... 页

`pageNo += total` 巧妙地实现了分页的分片，每个节点只处理自己负责的页码。

### 4.5 面试话术建议

> **面试官问**：签到功能和排行榜是怎么实现的？
>
> **回答**：签到功能我用了 **Redis BitMap**。Key 是 `sign:uid:{userId}:{yyyyMM}`，每个月一个 Key，每天对应一个 bit 位。`SETBIT` 操作设置签到标记，返回值为 0 说明是首次签到，为 1 说明重复签到。一个用户一个月的签到数据只占 4 字节，100 万用户也才 4MB。
>
> 计算连续签到天数用了 **BITFIELD GET** 一次性取出本月所有签到位，然后用位运算 `num & 1` 从最低位逐位检测，遇到 0 就停止，统计连续 1 的个数。
>
> 实时排行榜用 **Redis ZSet**，Member 是用户 ID，Score 是累计积分。每次新增积分用 `ZINCRBY` 原子递增。查排名用 `ZREVRANGE`（按分数倒序分页），查个人排名用 `ZREVRANK`，时间复杂度都是 O(log N)。
>
> 每月赛季结束后，需要将排行榜持久化到 MySQL 做历史查询。我用了 **动态分表**，每个赛季创建独立的表 `points_board_{seasonId}`。表名通过 **ThreadLocal** 传递，配合 MyBatis-Plus 的 `DynamicTableNameInnerInterceptor` 拦截器动态替换。
>
> 持久化任务用 **XxlJob 分片广播**，多个执行器节点并行处理不同的分页数据，通过 `ShardIndex` 和 `ShardTotal` 实现分页的分片跳跃，大幅提升数据迁移速度。

## 亮点五：优惠券兑换码算法（位运算 + 加权校验 + 双重混淆 + Base32编码）

### 5.1 设计背景与问题

优惠券兑换码需要满足以下要求：

- **全局唯一**：每个兑换码必须唯一，不能重复。
- **不可预测**：用户不能通过一个兑换码推测出其他有效兑换码。
- **可验证**：服务端能快速验证兑换码的合法性，拦截伪造的兑换码。
- **短小可读**：生成的兑换码要足够短（8-10个字符），方便人工输入。
- **可反解**：从兑换码中能解析出序列号，用于关联优惠券信息。

常见方案如 UUID 太长（32位）且无法反解，纯随机码需要数据库查重且无法校验。

### 5.2 解决方案

项目设计了一套精巧的**50位二进制兑换码算法**，整体结构如下：

```
兑换码明文结构（50位二进制）：
+--------+--------+------------------+
| 校验码  | 新鲜值  |     序列号        |
| 14位   |  4位   |     32位          |
+--------+--------+------------------+
   高14位   中4位       低32位

加密流程：
序列号(32位) + 新鲜值(4位)
       |
       v
  拼接载荷 payload = fresh << 32 | serialNum
       |
       v
  加权校验码 = Σ(payload每4位 × 加权表[i]) mod 2^14
       |
       v
  异或混淆：payload ^= XOR_TABLE[checkCode & 0b11111]
       |
       v
  拼接明文：checkCode(14位) | payload(36位)
       |
       v
  Base32编码 => 8~10位可读字符
```

### 5.3 核心代码分析

#### 5.3.1 生成兑换码

> 源码位置：`tj-promotion/src/main/java/com/tianji/promotion/utils/CodeUtil.java`（第103-116行）

```java
public static String generateCode(long serialNum, long fresh) {
    // 1.计算新鲜值（取优惠券ID的低4位）
    fresh = fresh & FRESH_MASK;   // FRESH_MASK = 0xF
    // 2.拼接payload：fresh(4位) + serialNum(32位) = 36位
    long payload = fresh << FRESH_BIT_OFFSET | serialNum;   // FRESH_BIT_OFFSET = 32
    // 3.计算加权校验码（14位）
    long checkCode = calcCheckCode(payload, (int) fresh);
    // 4.异或混淆：用校验码低5位做索引，从XOR_TABLE取密钥异或
    payload ^= XOR_TABLE[(int) (checkCode & 0b11111)];
    // 5.拼接兑换码明文：校验码(14位) + payload(36位) = 50位
    long code = checkCode << CHECK_CODE_BIT_OFFSET | payload;  // CHECK_CODE_BIT_OFFSET = 36
    // 6.Base32编码
    return Base32.encode(code);
}
```

#### 5.3.2 加权校验码计算

> 源码位置：`tj-promotion/src/main/java/com/tianji/promotion/utils/CodeUtil.java`（第118-129行）

```java
private static long calcCheckCode(long payload, int fresh) {
    // 1.根据新鲜值选择对应的加权码表（共16组）
    int[] table = PRIME_TABLE[fresh];
    // 2.payload每4位为一组，乘以对应的加权数，累加求和
    long sum = 0;
    int index = 0;
    while (payload > 0) {
        sum += (payload & 0xf) * table[index++];  // 取低4位 × 权重
        payload >>>= 4;                            // 右移4位
    }
    // 3.对2^14取模，得到14位校验码
    return sum & CHECK_CODE_MASK;   // CHECK_CODE_MASK = 0b11111111111111
}
```

**设计要点**：
- 36位 payload 被分为 9 组（每组4位），每组乘以 `PRIME_TABLE` 中的质数权重。
- **16组加权码表**由新鲜值 `fresh` 索引选择，同一张优惠券使用同一组码表，不同优惠券使用不同码表。
- 使用**质数**作为权重，降低碰撞概率。

#### 5.3.3 异或混淆密钥表

> 源码位置：`tj-promotion/src/main/java/com/tianji/promotion/utils/CodeUtil.java`（第41-50行）

```java
private final static long[] XOR_TABLE = {
```

**混淆逻辑**：`payload ^= XOR_TABLE[checkCode & 0b11111]`，用校验码的低5位（0-31）作为索引从 32 个大质数中选一个进行异或，打乱 payload 的位模式，使得相邻序列号生成的兑换码看起来完全不同。

#### 5.3.4 解析兑换码

> 源码位置：`tj-promotion/src/main/java/com/tianji/promotion/utils/CodeUtil.java`（第131-151行）

```java
public static long parseCode(String code) {
    if (code == null || !code.matches(RegexConstants.COUPON_CODE_PATTERN)) {
        throw new BadRequestException("无效兑换码");
    }
    // 1.Base32解码
    long num = Base32.decode(code);
    // 2.取低36位 => payload（混淆后的）
    long payload = num & PAYLOAD_MASK;       // PAYLOAD_MASK = 0xFFFFFFFFFL
    // 3.取高14位 => 校验码
    int checkCode = (int) (num >>> CHECK_CODE_BIT_OFFSET);
    // 4.用相同的密钥做异或，还原原始payload
    payload ^= XOR_TABLE[(checkCode & 0b11111)];
    // 5.从payload取高4位 => 新鲜值
    int fresh = (int) (payload >>> FRESH_BIT_OFFSET & FRESH_MASK);
    // 6.用还原后的payload重新计算校验码，与解码得到的校验码比较
    if (calcCheckCode(payload, fresh) != checkCode) {
        throw new BadRequestException("无效兑换码");
    }
    // 7.取低32位 => 序列号
    return payload & SERIAL_NUM_MASK;   // SERIAL_NUM_MASK = 0xFFFFFFFFL
}
```

#### 5.3.5 自定义 Base32 编码

> 源码位置：`tj-promotion/src/main/java/com/tianji/promotion/utils/Base32.java`（第8-19行）

```java
public class Base32 {
    // 自定义字符表，32个字符（排除了0、1、I、O等易混淆字符）
    private final static String baseChars = "6CSB7H8DAKXZF3N95RTMVUQG2YE4JWPL";

    public static String encode(long raw) {
        StrBuilder sb = new StrBuilder();
        while (raw != 0) {
            int i = (int) (raw & 0b11111);   // 取低5位
            sb.append(baseChars.charAt(i));
            raw = raw >>> 5;                  // 右移5位
        }
        return sb.toString();
    }

    public static long decode(String code) {
        long r = 0;
        char[] chars = code.toCharArray();
        for (int i = chars.length - 1; i >= 0; i--) {
            long n = baseChars.indexOf(chars[i]);
            r = r | (n << (5*i));
        }
        return r;
    }
}
```

**设计要点**：
- 字符表是打乱顺序的，增加了反向工程的难度。
- 50位二进制 / 5位每字符 = 10个字符，兑换码长度为 **8~10位**。
- 排除了 `0/1/I/O` 等易混淆字符，提升用户手工输入体验。

### 5.4 技术原理

**安全性分析**：

| 防护层 | 技术手段 | 作用 |
|---|---|---|
| 第1层 | 加权校验码 | 验证兑换码完整性，拦截随机伪造 |
| 第2层 | 16组加权码表 | 不同优惠券用不同码表，增加破解难度 |
| 第3层 | 异或混淆 | 打乱位模式，相邻序列号生成完全不同的兑换码 |
| 第4层 | 自定义Base32字符表 | 字符顺序打乱，增加反向工程难度 |
| 第5层 | 正则校验 | 格式预检，快速拦截明显非法输入 |

**空间效率**：
- 序列号 32 位，最多支持 **42 亿** 个兑换码。
- 校验码 14 位，碰撞概率为 1/16384。
- 整体 50 位，编码后仅 **10 个字符**，远短于 UUID 的 32 字符。

**反解机制**：
兑换码 -> Base32解码 -> 异或还原 -> 校验 -> 提取序列号 -> 通过 Redis ZSet（`COUPON_RANGE_KEY`）查找所属优惠券。

### 5.5 面试话术建议

> **面试官问**：优惠券兑换码是怎么生成的？怎么保证安全性？
>
> **回答**：我设计了一套基于位运算的兑换码生成算法。兑换码的明文是 50 位二进制数，分为三部分：**14位校验码 + 4位新鲜值 + 32位序列号**。
>
> 序列号是通过 Redis `INCR` 生成的全局唯一递增数字，保证不重复。新鲜值取自优惠券 ID 的低 4 位，同一张优惠券的所有兑换码共享一个新鲜值。
>
> 校验码的计算采用**加权求和**：将 36 位载荷（新鲜值 + 序列号）每 4 位为一组，分别乘以预设的质数权重表中的值，累加后对 2^14 取模。权重表有 16 组，由新鲜值索引选择，不同优惠券用不同码表。
>
> 为了防止相邻序列号生成相似的兑换码，我做了**异或混淆**：用校验码的低 5 位从 32 个大质数中选一个，与载荷做异或操作，彻底打乱位模式。
>
> 最后用**自定义 Base32 编码**将 50 位二进制转成 10 个可读字符，字符表排除了 0/1/I/O 等易混淆字符，还打乱了顺序。
>
> 解码时反向操作：Base32 解码、异或还原、重新计算校验码比对，如果不匹配直接拒绝。然后提取序列号，通过 Redis ZSet 查找所属优惠券。
>
> 这个方案的优势是：兑换码只有 10 个字符，方便输入；服务端可以纯计算验证合法性，不查数据库就能拦截伪造码；序列号支持 42 亿个，完全满足业务需求。
